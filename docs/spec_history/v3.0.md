## 1. Visão Geral e Caso de Negócio

### 1.1. Objetivo

Centralizar e padronizar o acompanhamento de projetos de consultoria da Jump Label através de uma plataforma executiva orientada por dados. O Gerente de Projetos (GP) alimenta as informações-chave de forma estruturada, e o sistema as transforma em visões estratégicas para cada perfil de usuário.

### 1.2. Pilares de inteligência

- **Ancoragem na Proposta.** Um agente lê as propostas comerciais e transforma seu conteúdo em baseline de escopo, prazo e entregas.
- **Inteligência Cruzada entre Projetos.** O sistema acumula dados históricos e cruza informações de múltiplos projetos.

### 1.3. Caso de negócio

| Premissa | Valor |
|---|---|
| Tempo economizado por GP por mês | 8 horas |
| Tempo economizado por projeto | 8 horas |
| Volume estimado em 12 meses | 30 projetos × 3 meses |
| Total de horas economizadas | 960 horas |
| Valor-hora de referência | R$ 100,00 |
| **Economia direta projetada** | **R$ 96.000,00 / ano** |

**Custo dos modelos:** zero por chamada (assinaturas Claude e ChatGPT já contratadas pela Jump). Custo fixo das assinaturas é tratado como custo do produto e documentado separadamente.

### 1.4. Piloto

Primeiro projeto cadastrado: **Bradesco — Migração SAS → Databricks**, em andamento. Operará em modo *shadow* na Fase 1 do agente leitor: extração automática é apresentada como sugestão; baseline só é ativado após revisão manual do GP.

## 2. Arquitetura Geral (Híbrida)

A solução opera em dois ambientes interligados:

```
┌─────────────────────────────────────┐         ┌──────────────────────────────────────┐
│      RAILWAY (cloud, 24/7)          │         │     MÁQUINA WINDOWS (Jump)           │
│                                     │         │                                      │
│  - Frontend Next.js                 │         │  jump-agent-runner                   │
│  - Backend FastAPI                  │   ◄──►  │  - Worker que puxa jobs da fila      │
│  - PostgreSQL                       │  jobs   │  - WSL2 + tmux + sessões persistent. │
│  - Redis (broker da fila)           │   via   │  - claude (-p) / claude broker       │
│  - Acesso de GPs, PMO, Cliente      │  Redis  │  - codex (exec) / codex broker       │
│                                     │         │  - Browser para login interativo     │
└─────────────────────────────────────┘         └──────────────────────────────────────┘
        Internet pública                              Atrás de NAT/firewall corporativo
```

**Por que essa divisão:**
- **Railway** carrega as cargas que precisam de disponibilidade 24/7 e acesso via internet (UI dos GPs, PMO e cliente Bradesco).
- **Máquina Windows local** carrega as cargas de IA, aproveitando as assinaturas Claude/ChatGPT já contratadas pela Jump, sem custo por chamada.

**Comunicação:** o backend nunca abre conexão de entrada para a máquina Windows. A máquina Windows **puxa** jobs de uma fila Redis no Railway. Resultados (JSON) são publicados de volta na fila ou diretamente em endpoint do backend autenticado por token.

### 2.1. Por que essa arquitetura, com olhos abertos

A escolha desta arquitetura foi feita após comparação explícita com alternativas (cloud puro com APIs, on-premises completo) e priorizou:

- **Custo zero por chamada de IA**, aproveitando assinaturas já contratadas
- **Privacidade ampliada** — propostas de clientes não viajam para APIs externas faturadas por uso, e podem ser processadas sob conta corporativa controlada
- **Resiliência via duas rotas** — para cada engine (Claude, Codex), há rota headless e rota broker, com fallback automático

As implicações operacionais decorrentes desta escolha estão na seção 8 e foram explicitamente aceitas.

## 5. Camada de Inteligência via `jump-agent-runner`

Esta é a mudança arquitetural central da v3. Onde antes havia um "LLM Gateway" que chamava APIs (Gemini → OpenAI → Claude), agora há o **`jump-agent-runner`**: uma camada de execução de agentes que invoca Claude Code e Codex CLI localmente, com duas rotas (headless e broker) por engine, fallback automático entre engines, e contrato de saída via arquivo JSON canônico.

### 5.1. Resumo da arquitetura do agent-runner

(Especificação técnica completa no documento `02_jump_agent_runner_spec.md`.)

```
Backend FastAPI ──[job na fila Redis]──► Worker Windows
                                                │
                                                ▼
                                         AgentRunner
                                                │
                              ┌─────────────────┴─────────────────┐
                              ▼                                   ▼
                       ClaudeProvider                      CodexProvider
                       (engine primário                    (engine fallback
                       quando configurado)                  quando primário falha)
                       headless → broker                   headless → broker
                              │                                   │
                              └─────────────┬─────────────────────┘
                                            ▼
                                     ArtifactValidator
                                  (JSON em output_path
                                   é a única fonte da verdade)
```

### 5.2. Política de fallback

- **Primeiro:** rota headless do engine primário
- **Se falhar** por login, quota, modo interativo, timeout sem artefato ou erro: rota broker do mesmo engine
- **Se broker falhar** sem produzir sentinel/artefato: engine secundário (mesma sequência)
- **API Key nunca é sugerida em nenhum nível** — não é parte da política

### 5.3. Tarefas dos agentes

Os três agentes do produto (descritos em detalhe na v2.1) são todos invocados via `jump-agent-runner`:

| Agente | Engine primário | Engine fallback | Justificativa da preferência |
|---|---|---|---|
| **Leitor de Propostas** (extração estruturada) | Claude | Codex | Claude tende a produzir JSON estruturado mais consistente |
| **Análise e Alertas** (insights por report) | Claude | Codex | Tarefa de análise e síntese — Claude é forte |
| **Inteligência Cruzada** (padrões de portfólio) | Claude | Codex | Idem |

Essas atribuições são **configuráveis** via `agent_runner_config.toml` no worker. A política do produto pode ser ajustada com base nos dados de `LLMCallLog` em produção (latência, taxa de sucesso, qualidade percebida).

### 5.4. Mudanças na entidade `LLMCallLog`

Renomeada para **`AgentRunLog`** para refletir o novo modelo. Campos:

| Campo | Descrição |
|---|---|
| `run_id` | ID único da execução, usado também como sentinel `AGENT_DONE:<run_id>` |
| `task_type` | `proposal_extraction`, `report_analysis`, `portfolio_pattern`, etc |
| `engine_used` | `claude` ou `codex` |
| `route_used` | `headless` ou `broker` |
| `attempts` | Lista JSONB com tentativas (engine, rota, sucesso, motivo de falha) |
| `duration_s` | Tempo total |
| `worker_id` | Qual máquina worker processou |
| `artifact_path` | Caminho do JSON aceito |
| `failover_occurred` | Booleano: true se houve mudança de engine |
| `created_at`, `completed_at` | |

`AgentRunLog` permite ao PMO ver no dashboard administrativo: quantos jobs foram pra Claude, quantos pra Codex (fallback), tempo médio, taxa de fallback. É a substituição do "custo por chamada" que tínhamos no v2 pelas métricas relevantes neste modelo (sucesso/fallback/latência).

## 6. Fluxo Principal do Sistema

```
┌──────────────────────────────────────────────────────────────────┐
│                       USUÁRIO (GP, PMO, Cliente)                 │
│                  acessa frontend hospedado em Railway             │
└──────────────────────────────┬───────────────────────────────────┘
                               │
                               ▼
┌──────────────────────────────────────────────────────────────────┐
│                  FRONTEND Next.js (Railway)                      │
└──────────────────────────────┬───────────────────────────────────┘
                               │ REST + SSE
                               ▼
┌──────────────────────────────────────────────────────────────────┐
│                  BACKEND FastAPI (Railway)                       │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Operações síncronas (CRUD, login, dashboards, leituras)  │   │
│  │ → respondem na hora                                      │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ Operações que precisam de IA (extração, análise)         │   │
│  │ → criam AgentRunLog, enfileiram em Redis, retornam 202   │   │
│  └──────────────────────────────────────────────────────────┘   │
└──────────────────────────────┬───────────────────────────────────┘
                               │
                               ▼
                      ┌────────────────┐
                      │   Redis Queue  │   (Railway gerencia)
                      │  (jobs.agent)  │
                      └────────┬───────┘
                               │
                               │ worker pulls
                               │
                               ▼
┌──────────────────────────────────────────────────────────────────┐
│              MÁQUINA WORKER WINDOWS (Jump, on-prem)              │
│                                                                  │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ jump-agent-runner-worker                                 │   │
│  │ - Conecta ao Redis do Railway via TLS                    │   │
│  │ - Puxa job (BLPOP), valida assinatura                    │   │
│  │ - Invoca AgentRunner com a tarefa                        │   │
│  │ - AgentRunner roda Claude/Codex via WSL+tmux             │   │
│  │ - Recebe artefato JSON canônico                          │   │
│  │ - Publica resultado via HTTP POST autenticado no backend │   │
│  └──────────────────────────────────────────────────────────┘   │
└──────────────────────────────┬───────────────────────────────────┘
                               │ resultado
                               ▼
                  Backend persiste e notifica
                  o usuário (SSE + email)
```

### 6.1. Fluxo de upload de proposta (exemplo concreto)

1. GP faz upload do PDF da proposta no frontend (Railway)
2. Backend salva o arquivo (R2/S3), cria `Proposal` com status `pending_extraction`
3. Backend cria `AgentRunLog` com `run_id` único, enfileira job em `jobs.agent` no Redis
4. Backend retorna HTTP 202 Accepted com `run_id` para o frontend
5. Frontend exibe estado "Extraindo baseline... (você será notificado)"
6. Worker Windows puxa o job, recebe URL do PDF, baixa para workspace local
7. AgentRunner monta `AgentTask` com `output_path` apontando para arquivo local
8. Tenta `claude -p` headless. Se falhar por motivo elegível, vai para broker. Se ainda falhar, fallback para Codex
9. Quando aceita um artefato JSON válido, publica resultado em `POST /internal/agent-results/<run_id>` no backend (autenticado por token shared-secret + assinatura HMAC)
10. Backend valida, persiste o `Baseline` e os `Deliverables` em status `draft`, atualiza `AgentRunLog`
11. Backend dispara notificação SSE/email para o GP: "Baseline pronto para revisão"
12. GP entra na tela de revisão, ajusta e ativa

Tempo típico esperado: 30 segundos a 3 minutos no caminho feliz; até 10 minutos com fallbacks completos.

### 6.2. O que o frontend nunca promete

- **Nunca** "extração imediata"
- **Nunca** spinner de carregamento aguardando síncrono por mais de 5 segundos
- **Sempre** comunicação assíncrona: "estamos processando, te avisaremos"

Isso é decisão de produto, não só técnica.

## 7. Arquitetura Técnica Detalhada

### 7.1. Componentes Railway

| Serviço | Tecnologia | Função |
|---|---|---|
| `frontend` | Next.js 14 + TypeScript + shadcn/ui | UI dos três perfis de usuário |
| `backend` | Python 3.12 + FastAPI + Pydantic v2 | API REST, autenticação, fluxo de aprovação |
| `postgres` | PostgreSQL 16 (Railway managed) | Persistência |
| `redis` | Redis (Railway managed) | Fila de jobs + cache |
| `notifications` | Resend (SMTP) + SSE no backend | Notificações |

### 7.2. Componentes da máquina Worker (Windows)

| Serviço | Tecnologia | Função |
|---|---|---|
| `jump-agent-runner` | Python 3.12 + asyncio | Biblioteca de execução de agentes |
| `worker` | Python (Celery em modo standalone, ou consumer Redis simples) | Puxa jobs e invoca o runner |
| `WSL2 + Ubuntu` | Microsoft + Canonical | Ambiente Linux para tmux e CLIs |
| `tmux` | apt | Sessões persistentes `project-claude` e `project-codex` |
| `claude` CLI | npm `@anthropic-ai/claude-code` | Engine Claude (headless e broker) |
| `codex` CLI | install.sh | Engine Codex (headless e broker) |

### 7.3. Comunicação Railway ↔ Worker

**Direção: Worker → Railway (sempre)**
- Worker conecta ao Redis do Railway via TLS, autenticado por senha
- Worker faz HTTP POST de resultados em `/internal/agent-results/<run_id>` autenticado com:
  - Token compartilhado (`WORKER_SHARED_SECRET`) no header
  - Assinatura HMAC-SHA256 do payload com chave `WORKER_HMAC_KEY`
  - Timestamp para evitar replay (janela de 5 minutos)

**Direção: Railway → Worker (nunca)**
- Backend nunca abre conexão TCP para a máquina worker
- Toda comunicação é iniciada pelo worker (NAT-friendly, firewall corporativo OK)

### 7.4. Modelo de fila

Uma única fila Redis: `jobs.agent`. Estrutura do job:

```json
{
  "run_id": "ext-prop-2026-05-06-001",
  "task_type": "proposal_extraction",
  "engine_primary": "claude",
  "engine_fallback": "codex",
  "input": {
    "file_url": "https://...",
    "file_sha256": "...",
    "project_id": "uuid",
    "language": "pt-BR"
  },
  "output_schema_url": "https://.../schemas/proposal_extraction.json",
  "result_callback_url": "https://api.../internal/agent-results/ext-prop-2026-05-06-001",
  "result_callback_token": "...",
  "timeout_hard_s": 600,
  "heartbeat_s": 30,
  "enqueued_at": "2026-05-06T14:23:00Z",
  "ttl_s": 14400
}
```

**Idempotência:** o `run_id` é a chave. Worker verifica antes de processar se já existe artefato e se o backend já recebeu resultado para aquele `run_id`. Reentregas da fila não geram processamento duplicado.

**TTL:** jobs com mais de 4h na fila são descartados e o `AgentRunLog` é marcado como `expired`. Notificação ao PMO.

**Dead-letter:** após 3 falhas consecutivas no mesmo `run_id`, o job vai para `jobs.agent.dead` e o PMO é notificado.

### 7.5. Modelo de dados (entidades novas/alteradas)

```
AgentRunLog (substitui LLMCallLog)
  run_id (PK), task_type, engine_used, route_used, attempts (JSONB),
  duration_s, worker_id, artifact_path, failover_occurred,
  status (queued/running/done/failed/expired), failure_reason,
  created_at, started_at, completed_at

WorkerHeartbeat (nova)
  worker_id (PK), last_seen_at, status, sessions_status (JSONB),
  jobs_processed_today, jobs_failed_today
```

`WorkerHeartbeat` permite ao PMO ver no dashboard administrativo: quais workers estão ativos, há quanto tempo, quantos jobs processaram.

## 8. Implicações Operacionais e Mitigações

Esta seção é nova na v3 e fundamental.

### 8.1. Disponibilidade da máquina worker

**Implicação:** se a máquina worker estiver desligada, todo upload de proposta fica preso na fila até ela voltar.

**Mitigações:**
- Máquina worker é **dedicada** (não laptop pessoal de ninguém), idealmente desktop ou mini-PC sempre ligado, conectado via cabo, em local com no-break
- `WorkerHeartbeat` no backend; se ausente por > 10 minutos em horário comercial, alerta automático para o operador via email
- UI honesta: GP que faz upload vê "fila atual: N jobs aguardando, tempo estimado X minutos"
- Política explícita: "fora do horário comercial, processamento pode demorar até a próxima manhã"

### 8.2. Governança das contas Claude/ChatGPT

**Implicação:** assinaturas são pessoais. Worker logado com conta pessoal cria dependência humana.

**Mitigações:**
- Criar **conta corporativa Jump** dedicada para cada provider:
  - `agentes@jump.com.br` (ou similar) com Claude Pro/Max
  - `agentes@jump.com.br` com ChatGPT Plus/Pro
- Custo das assinaturas é despesa do produto, não pessoal
- Senha das contas em cofre de senhas da Jump (1Password, Bitwarden), com mais de uma pessoa com acesso
- Quando um operador sai da Jump, a conta permanece; só a credencial de acesso ao cofre é revogada

**Custo estimado das assinaturas (referência, sujeito a verificação no momento da contratação):**
- Claude Pro: aproximadamente USD 20/mês
- ChatGPT Plus: aproximadamente USD 20/mês
- Total mensal: faixa de USD 40 a 50/mês conforme planos escolhidos

Esse custo é fixo e independe do volume — o caso de negócio do produto continua válido com folga (R$ 96k/ano de economia vs. R$ 3-4k/ano de assinaturas).

### 8.3. Re-login interativo

**Implicação:** assinaturas exigem login que pode expirar. Login do Claude/Codex CLI tipicamente abre browser.

**Mitigações:**
- Quando o broker detecta `LOGIN_REQUIRED`, ele envia o comando de login na sessão tmux e emite evento `user_login_required` para o backend
- Backend cria notificação no dashboard administrativo do PMO: "Worker X precisa de re-login do Claude. Operador, conecte na máquina e finalize"
- Operador acessa a máquina (presencial ou via Anydesk/RDP), executa `wsl tmux attach -t project-claude`, completa o login no browser, sai do tmux com Ctrl+B D
- Após sucesso, operador roda `jump-runner login-confirm claude` que reabilita o broker
- Job que disparou o login é re-enfileirado automaticamente

### 8.4. Latência percebida

**Implicação:** caminho feliz 30s-3min, fallback completo até 10min.

**Mitigações:**
- Frontend é **assíncrono honesto**, conforme detalhado na seção 6.2
- SSE para atualização em tempo real quando o resultado chega
- Email de notificação como fallback se o GP fechou o browser
- Estimativa de tempo na UI baseada em estatísticas reais (média de duração do `AgentRunLog` da última semana)

### 8.5. Crescimento (mais de uma máquina worker)

**Implicação:** uma máquina não sustenta indefinidamente.

**Mitigações:**
- Modelo já suporta **N workers** lendo da mesma fila Redis (BLPOP é atomic, sem dupla entrega)
- Cada worker registra `worker_id` único em `WorkerHeartbeat`
- Critério explícito de escalonamento: "quando média de tempo na fila exceder 10 minutos por 3 dias seguidos, adicionar segundo worker"

### 8.6. Plano de transição (saída para API se necessário)

**Critério de revisão:** após 3 meses de operação, comparar:
- Custo das assinaturas + custo operacional da máquina worker
- Versus: custo estimado de migrar para API (com volume real medido)
- Versus: tempo de operador gasto em re-logins, manutenção, monitoramento

Se a balança virar, há plano técnico:
1. `jump-agent-runner` ganha um terceiro provider: `APIProvider`
2. Configuração troca de `engine_primary=claude` (CLI) para `engine_primary=api`
3. Mesma interface, sem alterações em `AgentTask`/`AgentResult`
4. Migração gradual por tipo de tarefa

A decisão de hoje **não é irreversível**. Foi uma escolha consciente para o ponto atual da Jump, que pode ser revista com dados reais.

## 9. LGPD e Governança de Dados

A arquitetura híbrida exige diligência redobrada em LGPD. Esta seção documenta o tratamento.

### 9.1. Dados pessoais e sensíveis envolvidos

| Categoria | Origem | Onde repousa |
|---|---|---|
| Nome, email de usuários (GPs, PMO, cliente) | Cadastro | PostgreSQL Railway |
| Conteúdo de propostas comerciais | Upload | Storage R2 (cloud) + workspace temporário do worker |
| Conteúdo de reports (incluindo possíveis dados sensíveis de cliente) | Preenchimento dos GPs | PostgreSQL Railway |
| Logs de execução de IA (prompts e respostas) | Worker | Disco local da máquina worker + agregado em PostgreSQL |

### 9.2. Bases legais

- **Para dados de funcionários da Jump (GPs, PMO):** legítimo interesse para gestão da relação de trabalho
- **Para dados de clientes (Bradesco, futuros):** execução de contrato (cláusula contratual de prestação do serviço de consultoria)
- **Para dados pessoais que possam aparecer em propostas de terceiros:** depende do contrato com cada cliente; cláusula padrão a incluir nos contratos da Jump prevendo o tratamento via plataforma interna de gestão

### 9.3. Política de retenção

| Dado | Retenção |
|---|---|
| Propostas (originais) | 5 anos após encerramento do projeto |
| Reports | 5 anos após encerramento do projeto |
| Workspace temporário no worker (PDFs baixados, JSONs intermediários) | 7 dias |
| Logs do agent-runner (execução, prompts, sentinels) | 90 dias |
| `AgentRunLog` (metadados sem conteúdo do prompt/resposta) | 5 anos |

### 9.4. Cláusulas obrigatórias com providers de IA

Antes de o piloto Bradesco começar a enviar dados reais ao agente, a Jump confirma:

- **Anthropic Claude (assinatura Pro/Max):** confirmar nas configurações da conta o opt-out de uso de dados para treinamento. Anthropic publica essa configuração na conta do usuário.
- **OpenAI ChatGPT (Plus/Pro):** idem, confirmar o opt-out de "Improve the model for everyone" nas configurações de Data Controls.

Operador da máquina é treinado a verificar essas configurações em cada conta corporativa antes de habilitar o worker.

### 9.5. Trânsito de dados

Propostas trafegam:
1. Browser do GP → Backend Railway (HTTPS)
2. Backend Railway → R2 (HTTPS)
3. Worker Windows → R2 (HTTPS, baixa só quando há job)
4. Workspace temporário no worker → CLI do Claude/Codex (local, dentro de WSL)
5. Output JSON do CLI → Workspace → Backend Railway (HTTPS, autenticado HMAC)

Cada salto está sob TLS. Workspace local é em volume com permissões restritas ao usuário do worker. Limpeza automática após 7 dias.

### 9.6. Direitos do titular

- `GET /me/data-export`: exporta todos os dados do usuário autenticado em ZIP
- `POST /me/data-deletion-request`: cria pedido revisado pelo PMO
- Para dados de **clientes externos** (que não têm conta no sistema), pedidos de exercício de direito chegam via email institucional `lgpd@jump.com.br` e são atendidos manualmente em até 15 dias

### 9.7. Documentação obrigatória

`docs/lgpd.md` no repositório, atualizado e versionado, contendo:
- Controlador, encarregado (DPO)
- Operadores (Anthropic, OpenAI, Cloudflare/R2, Railway)
- Bases legais por finalidade

## 11. Métricas e Riscos

### 11.1. Métricas de produto
(Mantidas da v2.1)

| Métrica | Meta |
|---|---|
| Tempo médio para preencher um report | < 15 minutos |
| Taxa de adoção pelos GPs | > 80% |
| Precisão do agente leitor | > 85% sem edição manual |
| Reports liberados ao cliente no prazo | > 90% |

### 11.2. Métricas operacionais novas (v3)

| Métrica | Meta |
|---|---|
| Disponibilidade do worker em horário comercial | > 99% |
| Tempo médio em fila | < 2 minutos |
| Tempo médio de execução de tarefa | < 3 minutos |
| Taxa de fallback Claude → Codex | < 10% (se exceder, investigar) |
| Tarefas que precisaram de re-login interativo | < 1 por semana |

### 11.3. Riscos específicos da arquitetura

| Risco | Probabilidade | Mitigação |
|---|---|---|
| Máquina worker indisponível por hardware/queda de luz | Média | No-break, monitoramento, plano de segundo worker |
| Login expira em horário sem operador disponível | Média | Notificação imediata; SLA "atendimento até início do próximo dia útil" comunicado aos GPs |
| Quota da assinatura excedida | Baixa-Média | Fallback Claude→Codex; monitoramento de volume; possível upgrade de plano |
| Dados sensíveis vazam por má configuração de não-treinamento | Baixa | Procedimento documentado, checklist no setup, auditoria periódica |
| Custo das assinaturas sobe inesperadamente | Baixa | Plano de transição para API (seção 8.6) |
| Worker fica inalcançável do Railway por mudança de rede corporativa | Baixa | Worker reconecta automaticamente; logs de tentativa em CloudWatch/Sentry |
